# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A NixOS flake for two x86_64-linux machines: `unclebeam-pc` (AMD Ryzen 9 desktop + AMD RDNA GPU, gaming) and `unclebeam-thinkpad` (ThinkPad X1 Carbon Aura, Intel). Both run NixOS 26.05 + Hyprland (Sway was the original compositor, removed 2026-07 once Hyprland proved stable). The repo is edited primarily **on the Linux machines themselves**; the Mac is used as a remote install trigger (nixos-anywhere) and can only `nix eval` for validation — it can never build or switch.

## Commands

```sh
# Validate a config after edits (works cross-platform, incl. this Mac —
# evaluation only, no build). This is the main "did I break it" check:
nix eval .#nixosConfigurations.unclebeam-pc.config.system.build.toplevel.drvPath
nix eval .#nixosConfigurations.unclebeam-thinkpad.config.system.build.toplevel.drvPath

# Apply on a machine (hostname == flake attr, so no #name needed there):
sudo nixos-rebuild switch --flake .

# Smoke-test in a VM (Linux only):
nixos-rebuild build-vm --flake .#unclebeam-pc && ./result/bin/run-unclebeam-pc-vm

# Update pins (stays within the nixos-26.05 branch):
nix flake update
```

**⚠ Flakes only see git-tracked files.** After creating any new `.nix` file, `git add` it before evaluating/building — otherwise you get misleading "path does not exist" errors.

## Hard-won rules

These each caused a real failure at least once. Follow them literally.

- **A change is not "done" until it evals, is committed, AND is pushed — but a manual gate splits the work.** Half-committed or unvalidated work broke an install once — the machine pulled a state where only part of a change existed. So the work happens in two stages with the human's own rebuild in between. First: `nix eval` passes for BOTH hosts, then `git add .`, and **stop there — do not commit or push yet.** The human rebuilds and switches on their own machine first; only once they confirm it works do you commit, push, and print the commit sha. When wiring a new flake input, verify all three places it must appear: the input declared in `flake.nix`, its module wired into `mkHost`, and the import in the host that uses it.
- **Never invent hardware facts.** `hardware-configuration.nix` contents, `/dev/disk/by-id/` paths, filesystem UUIDs — these only come from the target machine. If one is missing, stop and ask the human; leave a clearly marked placeholder rather than guessing. A plausible-looking made-up value is worse than a loud gap, because disko will happily format the wrong disk.
- **Installs and reinstalls must pin an exact commit sha** — `github:unclebeam/nix-config/<sha>#<host>` — to bypass Nix's flake tarball caching (a branch ref can serve a cached tarball up to an hour stale, i.e. *not* the commit you just pushed). Branch refs are fine only for daily on-machine rebuilds with `--flake .`, where the repo is a local checkout and there's nothing to cache.

## Architecture

`flake.nix` defines a `mkHost` helper; each host is one `nixosConfigurations` entry whose modules are: the host dir, disko's NixOS module, and home-manager wired in as a NixOS module (`useGlobalPkgs`, so one `nixos-rebuild switch` builds system + user config together, and `allowUnfree` from core.nix applies to user packages too).

Three layers, each with a strict role:

- **`hosts/<name>/`** — deliberately THIN: hostname, hardware quirks (GPU, microcode, zram), `stateVersion`, and which shared modules to import. Importing a module *is* enabling it (e.g. the thinkpad gets gaming.nix by uncommenting one import line). Each host also has:
  - `hardware-configuration.nix` — machine-generated by `nixos-generate-config`; its `fileSystems.*` entries were removed because disko generates them.
  - `disko.nix` — declarative disk layout (GPT → 512M vfat ESP → btrfs with `@`/`@home` subvolumes, zstd compression). Dual purpose: nixos-anywhere uses it to **wipe and format** the disk at install time, and at run time it generates the mount config. Target disks are pinned by stable `/dev/disk/by-id/nvme-<model>_<serial>` paths, never `/dev/nvme0n1`.
- **`modules/`** — shared NixOS (system-level) modules: `core.nix` (nix settings, systemd-boot, NetworkManager, the user, baseline packages), `desktop.nix` (greetd/ReGreet greeter + GTK portal + fonts), `hyprland.nix` (hyprland session + uwsm + hyprlock PAM), `audio.nix`, `gaming.nix`, `laptop.nix`.
- **`home/`** — shared home-manager (user-level) config, identical on both machines; entrypoint `home/default.nix` imports hyprland/waybar/fuzzel/mako/alacritty/fish.

**System vs. user split for the desktop**: `modules/hyprland.nix` enables hyprland at the system level (session .desktop file, PAM for hyprlock, portals); the actual window-manager *configuration* lives in `home/hyprland.nix` plus the plain lua config `home/hypr/hyprland.lua` (symlinked, not Nix-generated — same rule as editor configs). Keep additions on the right side of that line: config-less system packages go in core.nix, apps with per-user config go in home/.

**One file per intent**: everything that exists *because of* one app lives in that app's file. If Dolphin needs Qt theming, both the package and the qt block belong in `home/dolphin.nix` — not a generic `qt.nix`. Removing an app must be atomic: delete one file plus its import line, leaving zero orphaned config behind. Only promote settings to a shared file when a second consumer actually appears, never preemptively.

**Theming**: `home/colors.nix` is the single source of truth for colors — a plain attrset (not a module) holding the melange dark palette, grouped `a` (UI), `b` (vivid), `c` (muted), `d` (dark accents). Consumers `import ./colors.nix` and interpolate values into their CSS/INI/TOML config strings. All values carry a leading `#`; strip it with `lib.removePrefix "#"` where a format demands (fuzzel, hyprlock, the generated `hypr/nix.lua`). Never hardcode a hex color in an app config.

**Invariants to preserve**:
- `nixosConfigurations` attribute names must equal each machine's `networking.hostName` — that's what makes bare `nixos-rebuild switch --flake .` work on-machine.
- `stateVersion` (system and home) is set once at first install; never bump it.
- Boot loader settings live in `core.nix`, *not* hardware-configuration.nix, so regenerating the hardware file can't drop them.
- home-manager's release branch must match the nixpkgs release when bumping either.

## Deliberate decisions (do not "fix")

Choices that look like gaps but are intentional. Don't "improve" them unprompted:

- **Editor configs stay plain files.** Neovim/Helix configs are ordinary config files symlinked via home-manager — never convert them to nixvim or other Nix-generated equivalents.
- **No Mason, ever.** LSPs and formatters are declared as Nix packages in `home/`, not installed by the editor.
- **No kanshi, no desktop-shell frameworks** (DMS/Noctalia/HyprPanel), **no extra compositors** — unless the human explicitly asks.
- **The restrained look is a feature, not an oversight.** Plain Waybar, melange palette. Hyprland deliberately carries small gaps, slight rounding, terminal blur, and fast animations (added on request, 2026-07) — and nothing more. Do not suggest further visual "improvements" unprompted.

## Style

Files are heavily commented in a teaching style — comments explain *why* a setting exists and what would break without it. Match that density and tone when editing.
