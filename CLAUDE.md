# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A NixOS flake for two x86_64-linux machines: `unclebeam-pc` (AMD Ryzen 9 desktop + AMD RDNA GPU, gaming) and `unclebeam-thinkpad` (ThinkPad X1 Carbon Aura, Intel). Both run NixOS 26.05 + niri (the compositor lineage is sway → hyprland → niri; each successor ran side-by-side as a trial, then the loser was deleted — hyprland was removed 2026-07). The repo is edited primarily **on the Linux machines themselves**; the Mac is used as a remote install trigger (nixos-anywhere) and can only `nix eval` for validation — it can never build or switch.

## Commands

```sh
# Validate a config after edits (works cross-platform, incl. this Mac —
# evaluation only, no build). This is the main "did I break it" check:
nix eval .#nixosConfigurations.unclebeam-pc.config.system.build.toplevel.drvPath
nix eval .#nixosConfigurations.unclebeam-thinkpad.config.system.build.toplevel.drvPath

# Apply on a machine (hostname == flake attr, so no #name needed there):
sudo nixos-rebuild switch --flake .

# Smoke-test in a VM (Linux only):
nixos-rebuild build-vm --flake .#unclebeam-pc && ./result/bin/run-unclebeam-pc-vm

# Update pins (stays within the nixos-26.05 branch):
nix flake update
```

**⚠ Flakes only see git-tracked files.** After creating any new `.nix` file, `git add` it before evaluating/building — otherwise you get misleading "path does not exist" errors.

## Hard-won rules

These each caused a real failure at least once. Follow them literally.

- **A change is not "done" until it evals, is committed, AND is pushed — but a manual gate splits the work.** Half-committed or unvalidated work broke an install once — the machine pulled a state where only part of a change existed. So the work happens in two stages with the human's own rebuild in between. First: `nix eval` passes for BOTH hosts, then `git add .`, and **stop there — do not commit or push yet.** The human rebuilds and switches on their own machine first; only once they confirm it works do you commit, push, and print the commit sha. When wiring a new flake input, verify all three places it must appear: the input declared in `flake.nix`, its module wired into `mkHost`, and the import in the host that uses it.
- **Never invent hardware facts.** `hardware-configuration.nix` contents, `/dev/disk/by-id/` paths, filesystem UUIDs — these only come from the target machine. If one is missing, stop and ask the human; leave a clearly marked placeholder rather than guessing. A plausible-looking made-up value is worse than a loud gap, because disko will happily format the wrong disk.
- **Installs and reinstalls must pin an exact commit sha** — `github:unclebeam/nix-config/<sha>#<host>` — to bypass Nix's flake tarball caching (a branch ref can serve a cached tarball up to an hour stale, i.e. *not* the commit you just pushed). Branch refs are fine only for daily on-machine rebuilds with `--flake .`, where the repo is a local checkout and there's nothing to cache.
- **Know which PAM service a display manager actually traverses.** When SDDM was the greeter, its PAM service was generated with `useDefaultRules = false` — a bare `substack login` — so per-service toggles like `sddm.kwallet`/`sddm.fprintAuth` were silent no-ops (one sat in niri.nix for months doing nothing; the keyring unlock actually came from a module hooking `login`). The current greeter runs on **greetd**, whose PAM service is generated WITH default rules and does NOT substack `login`: toggles on `security.pam.services.greetd` genuinely work, and toggles on `login` no longer fire at graphical login. (The DMS lock screen's *password* path does authenticate against `/etc/pam.d/login`, so that service still matters.) Before setting any per-service PAM option, check how that service is generated.

## Architecture

`flake.nix` defines a `mkHost` helper; each host is one `nixosConfigurations` entry whose modules are: the host dir, disko's NixOS module, and home-manager wired in as a NixOS module (`useGlobalPkgs`, so one `nixos-rebuild switch` builds system + user config together, and `allowUnfree` from core.nix applies to user packages too).

Three layers, each with a strict role:

- **`hosts/<name>/`** — deliberately THIN: hostname, hardware quirks (GPU, microcode, zram), `stateVersion`, and which shared modules to import. Importing a module *is* enabling it (e.g. the thinkpad gets gaming.nix by uncommenting one import line). Each host also has:
  - `hardware-configuration.nix` — machine-generated by `nixos-generate-config`; its `fileSystems.*` entries were removed because disko generates them.
  - `disko.nix` — declarative disk layout (GPT → 512M vfat ESP → btrfs with `@`/`@home` subvolumes, zstd compression). Dual purpose: nixos-anywhere uses it to **wipe and format** the disk at install time, and at run time it generates the mount config. Target disks are pinned by stable `/dev/disk/by-id/nvme-<model>_<serial>` paths, never `/dev/nvme0n1`.
- **`modules/`** — shared NixOS (system-level) modules: `core.nix` (nix settings, systemd-boot, NetworkManager, the user, baseline packages), `desktop.nix` (fonts + Wayland-wide env + fallback GTK portal), `niri.nix` (niri session + portal routing: KDE for dialogs, GNOME for capture), `dms.nix` (DMS system half: the flake module's service defaults + shell fonts), `dms-greeter.nix` (login greeter: DMS's greeter UI on greetd), `kwallet.nix` (session keyring: ksecretd + pam_kwallet auto-unlock on `greetd`; user half in `home/kwallet.nix`), `dolphin.nix` (avahi, udisks2, ntfs/exfat — the system half of the file manager), `audio.nix`, `gaming.nix`, `laptop.nix`.
- **`home/`** — shared home-manager (user-level) config, identical on both machines; entrypoint `home/default.nix` imports niri/dms/alacritty/fish.

**System vs. user split for the desktop**: `modules/niri.nix` enables niri at the system level (session .desktop file, portals, keyring); the actual window-manager *configuration* lives in `home/niri.nix` plus the plain KDL config `home/niri/config.kdl` (symlinked, not Nix-generated — same rule as editor configs). Keep additions on the right side of that line: config-less system packages go in core.nix, apps with per-user config go in home/.

**One file per intent**: everything that exists *because of* one app lives in that app's file. If Dolphin needs a system service, the app goes in `home/dolphin.nix` and the service in `modules/dolphin.nix` — not a generic file. Removing an app must be atomic: delete one file plus its import line, leaving zero orphaned config behind. Only promote settings to a shared file when a second consumer actually appears, never preemptively.

**The shell is DMS (DankMaterialShell); apps are KDE; screen capture is GNOME forever.** Since 2026-07, one quickshell process (the `dank-material-shell` flake input; `home/dms.nix` + `modules/dms.nix` + `modules/dms-greeter.nix`) is the bar, launcher, notification daemon, lock screen, idle policy, OSD, clipboard history, polkit agent, power menu, wallpaper, and greeter — it replaced waybar, fuzzel, swaync, gtklock, swayidle, plasma-polkit-agent, power.nix, and SDDM/SilentSDDM. Interaction is `dms ipc …` (the keybinds in config.kdl); configuration is the DMS Settings GUI, deliberately NOT the module's `settings` option (that would make the GUI read-only). Never run the shell twice (it runs as a user service; no `dms run` spawn in config.kdl). The KDE app layer stays: Dolphin + Ark + KIO, secrets in ksecretd (`modules/kwallet.nix` + `home/kwallet.nix`, auto-unlocked via pam_kwallet on `greetd` — which is why the greeter must stay password-only, see modules/fprintd.nix), dialogs/notifications/settings route to `xdg-desktop-portal-kde`. Do not re-add gnome-keyring. The one permanent exception: ScreenCast/Screenshot/RemoteDesktop stay routed to `xdg-desktop-portal-gnome` — niri's only capture path is `org.gnome.Mutter.ScreenCast`, while xdg-desktop-portal-kde's capture code is hard-wired to KWin's private `zkde_screencast` protocol and can never work here. Never route the capture family to `kde`.

**Theming**: DMS/matugen owns color. The shell derives a Material You palette from the wallpaper and (re)generates: `~/.config/niri/dms/colors.kdl` (included by config.kdl — niri's focus-ring/backdrop colors), `~/.config/alacritty/dank-theme.toml` (imported by alacritty.nix), GTK colors (via the installed `adw-gtk3` theme + the "Apply GTK Themes" toggle), and qt6ct/qt5ct color schemes (`home/qt.nix` sets `QT_QPA_PLATFORMTHEME=qt6ct`; Dolphin needs the KDE build, `kdePackages.qt6ct`). All of these are written imperatively by dms — configs reference them via includes/imports, never via `xdg.configFile` (a store symlink would block dms from writing them; `home/dms.nix`'s activation script guarantees empty placeholders before first run). There is no repo palette anymore (`home/colors.nix` was deleted with the migration); don't hardcode theme colors in app configs — let DMS reach them, or leave the app at its defaults (satty, mangohud).

**Invariants to preserve**:
- `nixosConfigurations` attribute names must equal each machine's `networking.hostName` — that's what makes bare `nixos-rebuild switch --flake .` work on-machine.
- `stateVersion` (system and home) is set once at first install; never bump it.
- The repo checkout lives at `~/nix-config` on every machine. `home/neovim.nix`'s out-of-store symlink hardcodes that path, and its failure mode is silent (a dangling `~/.config/nvim` — nvim just runs configless). Any future out-of-store symlink must use the same base path.
- Boot loader settings live in `core.nix`, *not* hardware-configuration.nix, so regenerating the hardware file can't drop them.
- home-manager's release branch must match the nixpkgs release when bumping either.

## Deliberate decisions (do not "fix")

Choices that look like gaps but are intentional. Don't "improve" them unprompted:

- **Editor configs stay plain files.** Neovim configs are ordinary config files symlinked via home-manager — never convert them to nixvim or other Nix-generated equivalents.
- **No Mason, ever.** LSPs and formatters are declared as Nix packages in `home/`, not installed by the editor.
- **DMS is the one shell — no second bar/launcher/locker/notification daemon** (no waybar, fuzzel, swaync, gtklock, swayidle, kanshi, other frameworks), **no extra compositors** — unless the human explicitly asks. Two notification daemons or two polkit agents on one bus is breakage, not choice.
- **DMS settings live in its GUI, not in Nix.** Don't populate `programs.dank-material-shell.settings` — a managed settings.json makes the DMS Settings GUI read-only.
- **`home/niri/config.kdl` deliberately TRACKS NIRI'S STOCK DEFAULTS** (the human is learning niri as designed). Deviations are limited to machine facts (monitors, keyboard layouts), session plumbing, and DMS plumbing (the `dms ipc` binds — the upstream DMS preset, with three displaced stock actions marked DISPLACED), each marked with a comment. Never port old hyprland habits (keybinds, gaps, animations, scratchpads) into it.

## Style

Files are heavily commented in a teaching style — comments explain *why* a setting exists and what would break without it. Match that density and tone when editing.
