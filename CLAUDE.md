# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A NixOS flake for two x86_64-linux machines: `unclebeam-pc` (AMD Ryzen 9 desktop + AMD RDNA GPU, gaming) and `unclebeam-thinkpad` (ThinkPad X1 Carbon Aura, Intel). Both run NixOS 26.05 + Hyprland 0.55+ with its **Lua config** (the compositor lineage is sway → hyprland → niri → hyprland again: this branch re-trials hyprland as a FULL replacement after it lost the first side-by-side trial and was deleted 2026-07-11; the niri setup lives in git history on `main`). The repo is edited primarily **on the Linux machines themselves**; the Mac is used as a remote install trigger (nixos-anywhere) and can only `nix eval` for validation — it can never build or switch.

## Commands

```sh
# Validate a config after edits (works cross-platform, incl. this Mac —
# evaluation only, no build). This is the main "did I break it" check:
nix eval .#nixosConfigurations.unclebeam-pc.config.system.build.toplevel.drvPath
nix eval .#nixosConfigurations.unclebeam-thinkpad.config.system.build.toplevel.drvPath

# Apply on a machine (hostname == flake attr, so no #name needed there):
sudo nixos-rebuild switch --flake .

# Smoke-test in a VM (Linux only):
nixos-rebuild build-vm --flake .#unclebeam-pc && ./result/bin/run-unclebeam-pc-vm

# Update pins (stays within the nixos-26.05 branch):
nix flake update
```

**⚠ Flakes only see git-tracked files.** After creating any new `.nix` file, `git add` it before evaluating/building — otherwise you get misleading "path does not exist" errors.

## Hard-won rules

These each caused a real failure at least once. Follow them literally.

- **A change is not "done" until it evals, is committed, AND is pushed — but a manual gate splits the work.** Half-committed or unvalidated work broke an install once — the machine pulled a state where only part of a change existed. So the work happens in two stages with the human's own rebuild in between. First: `nix eval` passes for BOTH hosts, then `git add .`, and **stop there — do not commit or push yet.** The human rebuilds and switches on their own machine first; only once they confirm it works do you commit, push, and print the commit sha. When wiring a new flake input, verify all three places it must appear: the input declared in `flake.nix`, its module wired into `mkHost`, and the import in the host that uses it.
- **Never invent hardware facts.** `hardware-configuration.nix` contents, `/dev/disk/by-id/` paths, filesystem UUIDs — these only come from the target machine. If one is missing, stop and ask the human; leave a clearly marked placeholder rather than guessing. A plausible-looking made-up value is worse than a loud gap, because disko will happily format the wrong disk.
- **Installs and reinstalls must pin an exact commit sha** — `github:unclebeam/nix-config/<sha>#<host>` — to bypass Nix's flake tarball caching (a branch ref can serve a cached tarball up to an hour stale, i.e. *not* the commit you just pushed). Branch refs are fine only for daily on-machine rebuilds with `--flake .`, where the repo is a local checkout and there's nothing to cache.
- **Know which PAM service a display manager actually traverses.** When SDDM was the greeter, its PAM service was generated with `useDefaultRules = false` — a bare `substack login` — so per-service toggles like `sddm.kwallet`/`sddm.fprintAuth` were silent no-ops (one sat in niri.nix for months doing nothing; the keyring unlock actually came from a module hooking `login`). The current greeter runs on **greetd**, whose PAM service is generated WITH default rules and does NOT substack `login`: toggles on `security.pam.services.greetd` genuinely work, and toggles on `login` no longer fire at graphical login. (The DMS lock screen's *password* path does authenticate against `/etc/pam.d/login`, so that service still matters.) Before setting any per-service PAM option, check how that service is generated.
- **Every session entry the greeter lists must actually boot.** The hyprland package unconditionally ships a second session file, `hyprland-uwsm.desktop`; with `withUWSM` off, uwsm's systemd user units don't exist, and landing on that entry killed logins in under a second (journal 2026-07-20: uwsm ran `systemctl --user start wayland-session-bindpid@….service` → exit status 5 → session closed → bounced back to the greeter — the "login bounces on first attempt" bug). `modules/hyprland.nix` force-filters `services.displayManager.sessionPackages` down to the plain `hyprland.desktop` via a link-farm (never `overrideAttrs` — that rebuilds hyprland from source). If another compositor/session ever appears, extend that filtered list; don't remove the filter.

## Architecture

`flake.nix` defines a `mkHost` helper; each host is one `nixosConfigurations` entry whose modules are: the host dir, disko's NixOS module, and home-manager wired in as a NixOS module (`useGlobalPkgs`, so one `nixos-rebuild switch` builds system + user config together, and `allowUnfree` from core.nix applies to user packages too).

Three layers, each with a strict role:

- **`hosts/<name>/`** — deliberately THIN: hostname, hardware quirks (GPU, microcode, zram), `stateVersion`, and which shared modules to import. Importing a module *is* enabling it (e.g. the thinkpad gets gaming.nix by uncommenting one import line). Each host also has:
  - `hardware-configuration.nix` — machine-generated by `nixos-generate-config`; its `fileSystems.*` entries were removed because disko generates them.
  - `disko.nix` — declarative disk layout (GPT → 512M vfat ESP → btrfs with `@`/`@home` subvolumes, zstd compression). Dual purpose: nixos-anywhere uses it to **wipe and format** the disk at install time, and at run time it generates the mount config. Target disks are pinned by stable `/dev/disk/by-id/nvme-<model>_<serial>` paths, never `/dev/nvme0n1`.
- **`modules/`** — shared NixOS (system-level) modules: `core.nix` (nix settings, systemd-boot, NetworkManager, the user, baseline packages), `nix-config.nix` (first-boot clone of the `~/nix-config` checkout — the target of every out-of-store symlink; no-op once it exists), `desktop.nix` (fonts + Wayland-wide env + fallback GTK portal), `hyprland.nix` (hyprland session + portal routing: KDE for dialogs, hyprland for capture), `dms.nix` (THE DMS shell: flake NixOS module enable + dms.service + shell fonts + input group), `dms-greeter.nix` (login greeter: DMS's greeter UI on greetd), `kwallet.nix` (session keyring: ksecretd + pam_kwallet auto-unlock on `greetd`; user half in `home/kwallet.nix`), `dolphin.nix` (avahi, udisks2, ntfs/exfat — the system half of the file manager), `audio.nix`, `gaming.nix`, `laptop.nix`.
- **`home/`** — shared home-manager (user-level) config, identical on both machines; entrypoint `home/default.nix` imports hyprland/dms/alacritty/fish.

**System vs. user split for the desktop**: `modules/hyprland.nix` enables hyprland at the system level (session .desktop file, portals, XWayland); the actual window-manager *configuration* lives in `home/hyprland.nix` plus the plain Lua config `home/hypr/hyprland.lua` (**out-of-store** symlinked into the `~/nix-config` checkout — same rule and same hardcoded base path as the editor configs, so edits apply on a bare `hyprctl reload` with no rebuild. The link must stay out-of-store: `dms setup` insists on writing that exact path and dies with EROFS against a store symlink — see the fresh-install flow below. hyprland 0.55+ loads hyprland.lua INSTEAD of hyprland.conf, and does not watch it — `hyprctl reload` after editing). Keep additions on the right side of that line: config-less system packages go in core.nix, apps with per-user config go in home/.

**One file per intent**: everything that exists *because of* one app lives in that app's file. If Dolphin needs a system service, the app goes in `home/dolphin.nix` and the service in `modules/dolphin.nix` — not a generic file. Removing an app must be atomic: delete one file plus its import line, leaving zero orphaned config behind. Only promote settings to a shared file when a second consumer actually appears, never preemptively.

**The shell is DMS (DankMaterialShell); apps are KDE; screen capture is the hyprland portal.** Since 2026-07, one quickshell process (the `dank-material-shell` flake input) is the bar, launcher, notification daemon, lock screen, idle policy, OSD, clipboard history, polkit agent, power menu, wallpaper, screenshot tool, and greeter — it replaced waybar, fuzzel, swaync, gtklock, swayidle, plasma-polkit-agent, power.nix, and SDDM/SilentSDDM. Upstream ships a NixOS module AND a home-manager module and documents choosing ONE — we use the **NixOS module only** (`modules/dms.nix`: enable + the one `dms.service` user unit + system packages + service defaults; `modules/dms-greeter.nix`: greeter; `home/dms.nix`: user-side glue ONLY — adw-gtk3 + the out-of-store `hypr/dms` symlink + the alacritty theme placeholder, it imports no DMS module). Never re-import the flake's homeModule: enabling both installs everything twice and risks two shells on one bus (upstream warns the shell must never run twice; also no `dms run` spawn in hyprland.lua). Interaction is `dms ipc …` and the keybinds live in the DMS-managed `~/.config/hypr/dms/binds.lua` (deployed by `dms setup`; git-tracked since 2026-07-20 via the out-of-store `hypr/dms` symlink, but still never hand-edited — the sole hand-written binds are the Print/screenshot→satty binds, a mandatory feature pinned in the DEVIATION block at the bottom of `home/hypr/hyprland.lua`); configuration is the DMS Settings GUI, deliberately NOT the module's `settings` option (that would make the GUI read-only). The seven `~/.config/hypr/dms/*.lua` fragments hyprland.lua require()s are DMS-owned and written imperatively (`dms setup` + the Settings GUI: Displays → outputs.lua, Keyboard Shortcuts → binds-user.lua, matugen → colors.lua) — never generate their content from Nix. Since 2026-07-20 the whole `~/.config/hypr/dms` dir is an out-of-store symlink to `home/hypr/dms/` (wired in `home/dms.nix`): writable, so dms's imperative writes keep working — they just land through the link in the git working tree (a read-only STORE symlink remains forbidden — dms would die with EROFS). Five of the seven fragments are git-tracked (binds/binds-user/layout/cursor/windowrules — shared preferences); tracking them replaced the old empty-placeholder activation script: a fresh clone already carries real content, so require() never hard-fails **and a fresh install has working keybinds from the first login**. The other two — **outputs.lua and colors.lua — are machine-derived and gitignored** (each host's monitors and wallpaper palette; syncing them would apply one machine's to the other and the two hosts' DMS writes would fight over the shared file): for those two, `home/dms.nix`'s activation script seeds empty placeholders through the symlink (an empty Lua chunk require()s fine — auto monitor config / default borders until DMS writes real content on that machine). A fresh install therefore boots to a WORKING desktop before any manual step: modules/nix-config.nix has already cloned the checkout, the tracked fragments provide real keybinds, and the greeter lists exactly one session. The manual first-login step remains for DMS's own state/settings: (1) `dms setup` — it refuses to touch fragments that already have content, so the tracked binds/layout survive — choosing compositor **Hyprland**, terminal **None** — NOT Alacritty: alacritty.toml is a read-only store symlink (generated from `programs.alacritty.settings`) and setup would die on it with EROFS the same way it once did on hyprland.lua; the dank-theme import is already wired in alacritty.nix so DMS needs nothing there — and systemd session **Yes**; (2) `cd ~/nix-config && git checkout -- home/hypr/hyprland.lua` — setup rewrites hyprland.lua with its autogenerated template *through* the out-of-store symlink, i.e. straight into the git working tree, and that rewrite must be discarded (our tracked template already require()s the fragments setup just deployed; `git status` shows the file modified until this step, and skipping it would let a later `git add .` stage DMS's template over ours); (3) `hyprctl reload`. Do NOT re-add automatic `dms setup` to the activation script: two attempts both broke fresh installs (the dms store path contains `=` which `env` parses as an assignment, and the CLI's privesc preflight fatals unless sudo is on PATH — activation's isn't); the CLI expects an interactive session, and the manual step is upstream's supported path. The KDE app layer stays: Dolphin + Ark + KIO, secrets in ksecretd (`modules/kwallet.nix` + `home/kwallet.nix`, auto-unlocked via pam_kwallet on `greetd` — which is why the greeter must stay password-only, see modules/fprintd.nix), dialogs/notifications/settings route to `xdg-desktop-portal-kde`. Do not re-add gnome-keyring. The exception: ScreenCast/Screenshot/GlobalShortcuts route to `xdg-desktop-portal-hyprland` (hyprland's native wlr-screencopy backend; RemoteDesktop is pinned to `none` — xdph doesn't implement it). Never route the capture family to `kde` — xdg-desktop-portal-kde's capture code is hard-wired to KWin's private `zkde_screencast` protocol and can never work here.

**Theming**: DMS/matugen owns color. The shell derives a Material You palette from the wallpaper and (re)generates: `~/.config/hypr/dms/colors.lua` (require()d by hyprland.lua — border/decoration colors), `~/.config/alacritty/dank-theme.toml` (imported by alacritty.nix), GTK colors (via the installed `adw-gtk3` theme + the "Apply GTK Themes" toggle), and qt6ct/qt5ct color schemes (`home/qt.nix` sets `QT_QPA_PLATFORMTHEME=qt6ct`; Dolphin needs the KDE build, `kdePackages.qt6ct`). All of these are written imperatively by dms — configs reference them via includes/imports, never via a store symlink (that would block dms from writing them). colors.lua and dank-theme.toml are machine-local and untracked (colors.lua is gitignored — each machine's wallpaper differs), both seeded as empty placeholders by `home/dms.nix`'s activation script before first run. There is no repo palette anymore (`home/colors.nix` was deleted with the migration); don't hardcode theme colors in app configs — let DMS reach them, or leave the app at its defaults (satty, mangohud).

**Invariants to preserve**:
- `nixosConfigurations` attribute names must equal each machine's `networking.hostName` — that's what makes bare `nixos-rebuild switch --flake .` work on-machine.
- `stateVersion` (system and home) is set once at first install; never bump it.
- The repo checkout lives at `~/nix-config` on every machine. `home/neovim.nix`'s out-of-store symlink hardcodes that path, and its failure mode is silent (a dangling `~/.config/nvim` — nvim just runs configless). Any future out-of-store symlink must use the same base path. `modules/nix-config.nix` auto-clones the checkout on the first boot that has network (ConditionPathExists makes it a permanent no-op afterwards), so a fresh nixos-anywhere install self-heals instead of booting with every symlink dangling.
- The session lifecycle is symmetric, and both halves live in `home/hypr/hyprland.lua`: the startup hook pushes the env and starts `hyprland-session.target` as ONE `sh -c '… && …'` command (two separate spawns raced — dms.service could start without `WAYLAND_DISPLAY`), and a `hyprland.shutdown` hook stops `graphical-session.target` on compositor exit (without it, logout left the user manager holding the dead session's env and the next login's shell came up broken). Don't remove either half, and keep them paired.
- Boot loader settings live in `core.nix`, *not* hardware-configuration.nix, so regenerating the hardware file can't drop them.
- home-manager's release branch must match the nixpkgs release when bumping either.

## Deliberate decisions (do not "fix")

Choices that look like gaps but are intentional. Don't "improve" them unprompted:

- **Editor configs stay plain files.** Neovim configs are ordinary config files symlinked via home-manager — never convert them to nixvim or other Nix-generated equivalents.
- **No Mason, ever.** LSPs and formatters are declared as Nix packages in `home/`, not installed by the editor.
- **DMS is the one shell — no second bar/launcher/locker/notification daemon** (no waybar, fuzzel, swaync, gtklock, swayidle, kanshi, other frameworks), **no extra compositors** — unless the human explicitly asks. Two notification daemons or two polkit agents on one bus is breakage, not choice.
- **DMS settings live in its GUI, not in Nix.** Don't populate `programs.dank-material-shell.settings` — a managed settings.json makes the DMS Settings GUI read-only.
- **`home/hypr/hyprland.lua` deliberately TRACKS DMS'S RECOMMENDED TEMPLATE** (the file `dms setup` would generate). Deviations are limited to machine facts (keyboard layouts — NOT monitors, those belong to the DMS Displays GUI via dms/outputs.lua), session plumbing, and mandatory features, each marked with a DEVIATION comment. Keybinds live in the DMS-managed dms/binds.lua + binds-user.lua (git-tracked under home/hypr/dms/, but written by DMS, never by hand) — except the Print/screenshot→satty binds, a mandatory feature that must survive a fresh install: they sit BELOW the require()s at the bottom of hyprland.lua (unbind-then-rebind; order is load-bearing). Never resurrect the pre-niri personalized hyprland config (scratchpads, groups, custom animations, melange palette) into it.

## Style

Files are heavily commented in a teaching style — comments explain *why* a setting exists and what would break without it. Match that density and tone when editing.
